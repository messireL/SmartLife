# SmartLife — пакет для переноса в новый чат

В этот пакет для нового чата входят:

- `docs/TRANSFER_TO_NEW_CHAT.md` — полный контекст проекта и хронология релизов;
- `docs/PORTAINER.md` — как сейчас разворачивать проект через Portainer;
- `docs/SERVER_COMMANDS.md` — полезные команды для текущего git/ssh и Portainer-сценария;
- `stack.env.portainer.example` — шаблон env для Portainer Stack;
- `docker-compose.portainer.yml` — актуальный Portainer compose.

## Что важно сообщить в новом чате

- текущая точка после этого релиза: `v0.11.31`;
- Portainer-стек больше не должен зависеть от `SMARTLIFE_APP_IMAGE`;
- в списке устройств теперь есть LAN-инвентаризация: локальный IP, MAC, метки `key получен`, `LAN-профиль`, `локальный режим`, `LAN ok`;
- при переносе существующей БД нужно использовать те же `APP_SECRET_KEY` и `DB_PASSWORD`;
- если Tuya Billing ещё не ожил, список устройств на новом пустом стенде не подтянется из облака автоматически;
- cloud-режим лучше держать в `Ручной`, чтобы не тратить лимит при его восстановлении.

- в `Синхронизация` появился JSON-резерв LAN-профилей и ключей: можно скачать backup, сохранить его на сервере и импортировать обратно без новых cloud-запросов к Tuya;
- для MAC добавлен безопасный fallback без лишнего расхода Tuya: поле в карточке устройства и массовый CSV-export/import LAN-инвентаризации;
- в задачах зафиксирован отдельный эксплуатационный блок: перевод SmartLife в Portainer/Docker на non-root контейнеры.


## v0.11.28
- локальный опрос состояний по LAN стал основным для устройств с подтверждённым LAN-профилем; cloud status теперь нужен только как fallback для остальных устройств;
- при `Tuya API = ручной` синхронизация не простаивает, а делает local-only refresh по LAN;
- в sync-результате видно, сколько снапшотов пришло локально, а сколько пришлось взять из cloud fallback.

## v0.11.30
- на вкладке `Локально` у устройства появился `LAN debug / сырые DP`: расплющенные локальные raw-поля, cloud statuses/definitions и pretty JSON последнего payload;
- это позволяет разбирать бойлеры и другие нестандартные Tuya-устройства по LAN без новых cloud-запросов и без повторного расхода лимита Tuya;
- UI показывает не только сырые поля, но и текущий маппинг SmartLife, чтобы быстро увидеть, куда ошибочно попала температура, режим или fault.

## v0.11.29
- SmartLife начал считать расход и стоимость не только по `energy_total_kwh`, но и по средней мощности между соседними локальными снапшотами, если устройство не отдаёт накопительный счётчик;
- список `Устройства` теперь показывает расход за сегодня и за месяц по каждому устройству и помечает, откуда он взялся: `счётчик устройства` или `расчёт по мощности`;
- это позволяет спокойно добирать телеметрию и тарифы по LAN без новых запросов в Tuya даже для power-only устройств.

## v0.11.31
- для бойлеров с локальным DP-мэппингом SmartLife больше не кладёт уставку в `fault`: `dps[10]` читается как текущая температура, `dps[9]` — как целевая температура, `dps[2]` — как режим;
- устройства, которые по LAN отдают только базовый статус без power/temp/energy, теперь явно помечаются как `status-only` / `LAN: статус без показаний`, чтобы было понятно: локальная связь есть, просто телеметрия у железки бедная.
