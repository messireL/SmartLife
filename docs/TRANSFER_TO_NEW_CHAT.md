# SmartLife — перенос в новый чат

## Проект
- Название: SmartLife
- Репозиторий: `https://github.com/messireL/SmartLife`
- Путь на сервере: `/opt/SmartLife`
- Стек: Python + FastAPI + PostgreSQL + Docker Compose
- Сервер работает в LAN, публикация обычно на `192.168.x.x:13443`
- Часовой пояс проекта: `Europe/Moscow`

## Последняя подтверждаемая точка в архиве
- База для этого релиза: `v0.11.31`
- Новый релиз в этом архиве: `v0.11.35`
- Основное изменение: container healthcheck в обычном compose и Portainer остаётся на лёгком `/favicon.ico`, чтобы получение local key / LAN-probe не провоцировали перезапуски приложения из-за тяжёлого `/health`.
- Основное изменение: успешный LAN-probe больше не включает молча `Включить LAN-профиль` и `Предпочитать LAN`; SmartLife сохраняет найденные IP / protocol version, а политику оставляет пользователю.
- Основное изменение: Tuya Cloud IP используется как LAN-адрес только если он действительно локальный/private; внешний IP больше не затирает ручной локальный адрес и не запускает заведомо бесполезный probe.
- Основное изменение: список `Устройства` показывает LAN-инвентаризацию по каждому девайсу: локальный IP, MAC, метки `key получен`, `LAN-профиль`, `локальный режим`, `LAN ok` и время последнего ручного получения key.
- Основное изменение: в `Синхронизация` добавлен JSON-резерв локального контура — можно скачать backup LAN-профилей и ключей, сохранить его на сервере и импортировать обратно без повторного запроса key из Tuya.
- Основное изменение: MAC теперь можно заполнять и без Tuya/ARP-магии: вручную в карточке устройства или пачкой через CSV-импорт/экспорт на странице `Синхронизация`.

## Как работаем
1. Сначала делается релиз/архив.
2. Commit message даётся отдельно.
3. Пользователь пушит через GitHub Desktop.
4. Сервер обновляется **только через git**: `git pull --ff-only`, затем `./scripts/manage.sh up`. Скрипт сам делает rebuild по умолчанию.
5. Не накатывать архив поверх git-репозитория как основной сценарий.

## Что уже есть
- Tuya Cloud интеграция
- Фоновая синхронизация
- Графики и дашборд
- Комнаты и локальные имена устройств
- Бэкапы БД
- Управление устройствами через `switch`, `switch_1`, а также поддержка бойлера через `mode` и `temp_set`
- Cloud-настройки хранятся в PostgreSQL (`app_settings`)
- Тариф электроэнергии поддерживает единый, двухзонный и трёхзонный режимы по времени суток
- В разделе Настройки есть веб-форма изменения cloud-настроек
- `docs/SERVER_COMMANDS.md` содержит актуальные команды для сервера
- `./scripts/manage.sh runtime-info` показывает диагностику runtime, тарифного месяца и схемы БД

## Важные правила
- В каждый релиз класть и обновлять этот файл и `docs/SERVER_COMMANDS.md`.
- В обычных командах `health` и `url` не чистить экран.
- `./scripts/manage.sh up` теперь по умолчанию пересобирает контейнеры; для редкого запуска без rebuild есть `./scripts/manage.sh up --no-build`.
- Очистку stopped-контейнеров делать только для mutating-команд или отдельно через `cleanup-docker`.
- demo-устройства можно чистить, обычные устройства не удалять из UI.

## Обновление v0.11.32
- regular local sync больше не перебирает все версии TinyTuya по каждому устройству: SmartLife использует уже сохранённую рабочую версию протокола и короткий timeout, поэтому одно проблемное устройство не должно тянуть всю синхронизацию на много минут;
- ручной `LAN-probe` остаётся отдельным действием и по-прежнему умеет подбирать версию протокола, но это больше не происходит внутри каждого обычного sync;
- для бойлера уточнён LAN-мэппинг: `dps[10]` — текущая температура, `dps[9]` — целевая температура, `dps[2]` — режим; ложный `fault`, совпадавший с уставкой, больше не должен появляться после нового sync;
- устройства, у которых по LAN видны характерные metering-DP (`17/18/19/20`), теперь распознаются как профиль `розетка с измерением`, а не как безликий `status-only`;
- в задачах по-прежнему отдельным блоком держится перевод SmartLife в Portainer/Docker на non-root контейнеры.

## HTTP / HTTPS

Сейчас SmartLife публикуется в LAN по HTTP, потому что стек поднимает приложение напрямую на локальном IP и порту без отдельного reverse proxy и TLS-сертификата. Для HTTPS нужен отдельный слой терминатора TLS (например, Caddy или Nginx) и локальный сертификат/доверенный CA.
