# SmartLife — перенос в новый чат

## Проект
- Название: SmartLife
- Репозиторий: `https://github.com/messireL/SmartLife`
- Путь на сервере: `/opt/SmartLife`
- Стек: Python + FastAPI + PostgreSQL + Docker Compose
- Сервер работает в LAN, публикация обычно на `192.168.x.x:13443`
- Часовой пояс проекта: `Europe/Moscow`

## Последняя подтверждаемая точка в архиве
- База для этого релиза: `v0.11.22`
- Новый релиз в этом архиве: `v0.11.23`
- Основное изменение: Portainer-стек больше не зависит от `SMARTLIFE_APP_IMAGE`; сервис `app` собирается напрямую из Git через `build`, чтобы новые релизы не залипали на старом image tag.
- Основное изменение: в docs обновлён сценарий переноса в Portainer и подготовлен актуальный пакет файлов для переноса проекта в новый чат.

## Как работаем
1. Сначала делается релиз/архив.
2. Commit message даётся отдельно.
3. Пользователь пушит через GitHub Desktop.
4. Сервер обновляется **только через git**: `git pull --ff-only`, затем `./scripts/manage.sh up`. Скрипт сам делает rebuild по умолчанию.
5. Не накатывать архив поверх git-репозитория как основной сценарий.

## Что уже есть
- Tuya Cloud интеграция
- Фоновая синхронизация
- Графики и дашборд
- Комнаты и локальные имена устройств
- Бэкапы БД
- Управление устройствами через `switch`, `switch_1`, а также поддержка бойлера через `mode` и `temp_set`
- Cloud-настройки хранятся в PostgreSQL (`app_settings`)
- Тариф электроэнергии поддерживает единый, двухзонный и трёхзонный режимы по времени суток
- В разделе Настройки есть веб-форма изменения cloud-настроек
- `docs/SERVER_COMMANDS.md` содержит актуальные команды для сервера
- `./scripts/manage.sh runtime-info` показывает диагностику runtime, тарифного месяца и схемы БД

## Важные правила
- В каждый релиз класть и обновлять этот файл и `docs/SERVER_COMMANDS.md`.
- В обычных командах `health` и `url` не чистить экран.
- `./scripts/manage.sh up` теперь по умолчанию пересобирает контейнеры; для редкого запуска без rebuild есть `./scripts/manage.sh up --no-build`.
- Очистку stopped-контейнеров делать только для mutating-команд или отдельно через `cleanup-docker`.
- demo-устройства можно чистить, обычные устройства не удалять из UI.

## HTTP / HTTPS

Сейчас SmartLife публикуется в LAN по HTTP, потому что стек поднимает приложение напрямую на локальном IP и порту без отдельного reverse proxy и TLS-сертификата. Для HTTPS нужен отдельный слой терминатора TLS (например, Caddy или Nginx) и локальный сертификат/доверенный CA.

- страница `Синхронизация` теперь показывает статус квоты Tuya Trial и время/источник последнего сигнала о лимите;
- версия поднята до `v0.11.16`, потому что после уже собранного `v0.11.15` появились дополнительные изменения кода и нумерацию нужно держать честно.



## Подготовка к Portainer

- в проект добавлен `docker-compose.portainer.yml` для деплоя из Git через Portainer Stack;
- добавлен `stack.env.portainer.example`;
- приложение умеет читать `SMARTLIFE_APP_SECRET_KEY` и `SMARTLIFE_DB_PASSWORD` из environment, чтобы не требовать файловых secrets на хосте;
- основной старый сценарий через `scripts/manage.sh` и `docker-compose.yml` сохранён.


## Обновление v0.11.23

- Portainer-стек больше не использует `SMARTLIFE_APP_IMAGE`;
- сервис `app` в `docker-compose.portainer.yml` собирается напрямую из Git через `build`, чтобы релизы не залипали на старом образе;
- обновлены `docs/PORTAINER.md`, `docs/SERVER_COMMANDS.md` и этот файл переноса.
