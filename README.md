# SmartLife

SmartLife — веб-приложение для управления устройствами Smart Life / Mi Home, просмотра статусов, расхода электроэнергии и базового управления розетками и многоканальными устройствами.

## Что нового в v0.11.21
- добавлен ручной режим Tuya API: автоматический cloud sync полностью выключается, а при восстановлении облака `local_key` и `IP` подтягиваются только ручным запросом по нужному устройству;
- ручной fetch по устройству остаётся точечным: SmartLife использует endpoint карточки устройства, сохраняет `local_key`/`ip` и дальше работает через LAN без повседневного опроса облака;

Стек:
- Python 3.12
- FastAPI + Jinja2
- PostgreSQL
- Docker Compose

## Что есть в v0.11.21
- в `v0.11.20` SmartLife по умолчанию включает `Предпочитать LAN` для новых/ещё не настроенных LAN-профилей, а при CSV-импорте применяет `Prefer LAN`, если колонка `prefer_local` отсутствует или пустая;
- на странице `Синхронизация` появился LAN-блок для массовой работы: можно загрузить CSV с `external_id/local_ip/protocol_version/local_key` и пачкой обновить LAN-профили устройств;
- там же появился пакетный `LAN probe / status-check`: SmartLife проходит по полным LAN-профилям, перепроверяет локальную доступность и при успехе фиксирует рабочую версию протокола в карточке устройства;
- импорт CSV не затирает уже сохранённый `local_key`, если колонка пустая; для явной очистки предусмотрена колонка `clear_local_key=true`;
- в статистике страницы `Синхронизация` теперь видно, сколько устройств уже имеют LAN-профили, сколько из них готовы к LAN и сколько успешно проходили последний probe;

## Что есть в v0.11.18
- в карточке Tuya-устройства появилась кнопка `Получить key из Tuya и проверить LAN`: один ручной cloud-запрос по `device_id` подтягивает `local_key` и `ip`, а затем SmartLife сразу пытается подобрать рабочую версию локального протокола;
- после успешного fetch+probe SmartLife сам включает LAN-профиль и `Предпочитать LAN`, чтобы для switch-команд больше не жить на постоянном cloud-контуре;
- `sync` больше не устраивает трагедию с traceback при исчерпанной Tuya Trial quota: CLI и веб-запуск теперь переводят цикл в `degraded skip`, пишут понятный статус и оставляют кэш устройств в покое;
- в карточке устройства на вкладке `Локально` появился ручной LAN-профиль Tuya: можно сохранить `local_ip`, `protocol_version`, `local_key`, а также включить режим `предпочитать LAN`;
- для Tuya-устройств со switch-каналами SmartLife теперь умеет использовать сохранённый LAN-профиль как fallback при exhausted quota, а при включённом `предпочитать LAN` — и без ожидания облака;
- текущий LAN-мост в `v0.11.17` поддерживает именно switch-команды (`switch`, `switch_1`, `switch_2`, ...), то есть розетки, бойлеры с `switch_1` и каналы удлинителей; режимы, температуры и прочая экзотика пока остаются на облаке;
- на странице `Синхронизация` добавлен счётчик `Пропущено`, чтобы degraded-cycle не маскировался под ошибку или успех.

## Быстрый старт

### Новый сервер

```bash
cd /opt
git clone https://github.com/messireL/SmartLife.git SmartLife
cd /opt/SmartLife
chmod +x scripts/manage.sh
./scripts/manage.sh up
./scripts/manage.sh seed-demo
./scripts/manage.sh health
./scripts/manage.sh runtime-info
./scripts/manage.sh url
```

### Обновление уже развёрнутого сервера

```bash
cd /opt/SmartLife
git pull --ff-only
chmod +x scripts/manage.sh
./scripts/manage.sh up
./scripts/manage.sh health
./scripts/manage.sh runtime-info
./scripts/manage.sh url
```

### Развёртывание через Portainer

- использовать `docker-compose.portainer.yml`;
- пример переменных среды лежит в `stack.env.portainer.example`;
- `SMARTLIFE_APP_IMAGE` больше не нужен: Portainer-стек собирает `app` напрямую из Git через `build`, чтобы релизы не залипали на старом image tag;
- отдельная инструкция лежит в `docs/PORTAINER.md`;
- для Portainer приложение умеет читать `SMARTLIFE_APP_SECRET_KEY` и `SMARTLIFE_DB_PASSWORD` прямо из environment без файлов `secrets/`.


## Подключение Tuya Cloud

```bash
cd /opt/SmartLife
./scripts/manage.sh configure-tuya

./scripts/manage.sh configure-timezone Europe/Moscow
./scripts/manage.sh up
./scripts/manage.sh health
./scripts/manage.sh runtime-info
./scripts/manage.sh url
```

Ручная синхронизация остаётся доступной:

```bash
./scripts/manage.sh sync
```

## Резервные копии базы

Автобэкап создаётся перед:
- `./scripts/manage.sh restart`
- `./scripts/manage.sh up`

Ручные команды:

```bash
./scripts/manage.sh backup-db
./scripts/manage.sh backup-list
./scripts/manage.sh backup-prune [keep_last]
./scripts/manage.sh restore-db backups/db/<file>.dump
./scripts/manage.sh cleanup-demo
```

Важно:
- автосохранение включено;
- **автовосстановление не включено специально**, чтобы не откатывать базу назад без явной команды.

## Часовой пояс и время

- хранение времени остаётся в UTC;
- UI и day/month агрегаты используют `SMARTLIFE_TIMEZONE`;
- по умолчанию — `Europe/Moscow`;
- формат вывода времени в UI: `ДД-ММ-ГГГГ ЧЧ:ММ:СС`.

## Где лежат секреты

```text
/opt/SmartLife/secrets/
```

Основные файлы:
- `app_secret_key`
- `db_password`
- `smartlife_xiaomi_username`
- `smartlife_xiaomi_password`
- `smartlife_xiaomi_device_token`

Tuya Cloud settings теперь хранятся в PostgreSQL (`app_settings`). Старые файлы `smartlife_tuya_*` больше не нужны для новых установок и используются только как legacy-источник для миграции старых стендов.

## HTTP / HTTPS

Сейчас SmartLife публикуется в LAN по HTTP, потому что стек поднимает приложение напрямую на локальном IP и порту без отдельного reverse proxy и TLS-сертификата. Для HTTPS нужен отдельный слой терминатора TLS (например, Caddy или Nginx) и локальный сертификат/доверенный CA.


- Hotfix: восстановлена совместимость версии приложения (`APP_VERSION`) и устранён регресс, из-за которого контейнер `app` не стартовал после v0.9.5.

## Power strip / удлинитель

В v0.10.2 SmartLife понимает устройства Tuya, у которых есть несколько каналов `switch_1...switch_N` и USB-блок `switch_usb` / `switch_usbN`.
В карточке такого устройства показываются отдельные каналы питания, а на вкладке управления можно включать и выключать каждую линию отдельно.

## Новое в v0.10.6

- для удлинителей и сетевых фильтров Tuya добавлены расширенные команды: `countdown_N`, `relay_status`, `light_mode`, `child_lock`;
- на вкладке `Управление` видно текущее состояние этих параметров и можно менять их прямо из SmartLife;
- countdown выводится по каждому каналу отдельно, с текущим значением и ограничениями Tuya;
- после `sync` SmartLife подтягивает definitions из Tuya и использует их для валидации enum/int-команд.


## Обновление v0.10.7
